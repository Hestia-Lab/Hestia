# @hestia/contracts

On-chain pool, association-set registry, and generated Groth16 verifiers. **Status: built + tested (P3).**
Foundry; Solidity ^0.8.24; target Base. Non-custodial — no role can move user funds.

```
src/
  HestiaPool.sol              # proof-less shield + transact (1x2/2x2), nullifiers, ETH+USDC vault
  MerkleTreeWithHistory.sol   # incremental Poseidon tree (depth 32, 64-root window)
  AssociationSetRegistry.sol  # multi-ASP approved roots (compliance gate)
  interfaces/                 # IPoseidon, IVerifier, IERC20
  verifiers/                  # TransactionVerifier1x2/2x2 (generated from circuits, gitignored-safe)
script/
  gen-poseidon.mjs            # circomlib -> Poseidon bytecode (T2/T3/T6) for on-chain hashing
  Deploy.s.sol                # Poseidon -> verifiers -> registry -> pool
test/                         # Poseidon/Merkle conformance, full shield->unshield e2e with a real proof
```

## Prerequisites & build

```bash
curl -L https://foundry.paradigm.xyz | bash && ~/.foundry/bin/foundryup   # forge/cast/anvil
pnpm --filter @hestia/contracts gen:poseidon                              # Poseidon bytecode (needs circomlibjs)
forge build
forge test          # 11 tests: on-chain Poseidon/Merkle == @hestia/common, e2e unshield with a Groth16 proof
```

The e2e proof fixture is produced by the circuits package:
`pnpm --filter @hestia/circuits exec node scripts/gen-proof-fixture.mjs` →
`test/fixtures/unshieldEth1x2.json`. On-chain Poseidon is the circomlib-generated bytecode, so the
contract's commitments and Merkle root are byte-identical to the SDK and circuits.

## Deploy

```bash
forge script script/Deploy.s.sol:Deploy                                   # local simulation
forge script script/Deploy.s.sol:Deploy \
  --rpc-url base_sepolia --broadcast --private-key $PRIVATE_KEY            # Base Sepolia
USDC_ADDRESS=0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913 \
forge script script/Deploy.s.sol:Deploy \
  --rpc-url base --broadcast --private-key $PRIVATE_KEY --slow             # Base mainnet
```

Set RPC / key / overrides in `.env` (see `.env.example`). The actual testnet/mainnet broadcast
needs a funded deployer key — it is the one P3 step that requires secrets. The deploy order and
wiring are deterministic; afterwards, confirm the pool's `immutable`s on-chain (e.g.
`cast call $POOL 'usdc()(address)'`) match the intended USDC / verifiers / registry.

### Production verifiers (mainnet)

The verifiers checked into `src/verifiers/` are the **dev** ceremony's — reproducible and
intentionally insecure (public toxic waste). A real deployment must not use them. Generate a
fresh, discarded-entropy setup and swap it in only for the broadcast:

```bash
node ../circuits/scripts/ceremony-prod.mjs        # → ../circuits/build/prod/ (gitignored)
#   writes fresh TransactionVerifier{1x2,2x2}.sol + *_final.zkey; entropy is from
#   crypto.randomBytes and never persisted, so the toxic waste is discarded.
cp ../circuits/build/prod/TransactionVerifier*.sol src/verifiers/   # swap in for the build
forge build && forge script ... --broadcast                         # deploy (see above)
git checkout -- src/verifiers/  # (or restore the dev verifiers) — keep the repo on the dev set
```

The production `zkey`s are what the client must prove against; copy them to wherever the app
serves circuit artifacts (in this product, `hestia/public/circuits/*.zkey`). The setup is
single-contributor — a public multi-party ceremony is still the bar before large value.
